# One project. Claude Code and Codex.

<div className={styles.page}>

A plain-English guide to setting up your project so Claude Code and Codex can both work from the same context. The same pattern can support other coding agents too.

## 1. The problem in 30 seconds

Every AI coding agent needs instructions, reusable workflows, and tool settings. Each tool names those files differently.

<div className={styles.explainer}>
  <p><strong>Claude Code</strong> looks for <code>CLAUDE.md</code> and <code>.claude/</code>.</p>
  <p><strong>Codex</strong> looks for <code>AGENTS.md</code>, <code>.codex/</code>, and <code>.agents/skills/</code>.</p>
  <p><strong>The fix:</strong> give each tool the files it expects, but keep your real project knowledge in shared folders like <strong>docs/</strong>, <strong>references/</strong>, and <strong>templates/</strong>.</p>
</div>

## 5. Five things that trip up beginners

Worth reading before you start. These are the differences you can't paper over.

<div className={styles.gotchaGrid}>
  <div className={styles.gotcha}>
    <h4>1. Both tools have skills, in different folders.</h4>
    <p>Claude reads from <code>.claude/skills/</code>. Codex reads from <code>.agents/skills/</code>. Same basic <code>SKILL.md</code> shape, but Claude-specific tool names or hooks may need small edits for Codex.</p>
  </div>
  <div className={styles.gotcha}>
    <h4>2. Different formats for agents.</h4>
    <p>A Claude agent is a <code>.md</code> file with a small YAML header. A Codex agent is a <code>.toml</code> file. The instructions inside are the same idea, just wrapped differently.</p>
  </div>
  <div className={styles.gotcha}>
    <h4>3. Settings files are not interchangeable.</h4>
    <p>You can't copy <code>.claude/settings.json</code> into <code>.codex/config.toml</code>. They're different formats and configure different things. Keep them separate.</p>
  </div>
  <div className={styles.gotcha}>
    <h4>4. Codex reads nested instructions.</h4>
    <p>Both tools support a CLAUDE.md or AGENTS.md inside a subfolder for scoped instructions. Codex chains them from the repo root down to your current directory. The closer file wins.</p>
  </div>
  <div className={styles.gotcha}>
    <h4>5. Sub-agents work differently.</h4>
    <p>Claude agents are markdown files. Codex agents are TOML files. The role prompt can be similar, but the wrapper and execution model are different.</p>
  </div>
  <div className={styles.gotcha}>
    <h4>Bonus: keep shared context out of tool folders.</h4>
    <p>Put reusable knowledge in <code>docs/</code>, <code>references/</code>, and <code>templates/</code>. Then both tools can read the same source instead of drifting apart.</p>
  </div>
</div>

## 6. The tool-agnostic setup

Split your project into three layers. Shared knowledge in one place, workflows in another, tool-specific config in their own folders.

<div className={styles.solution}>
  <h3>The three-layer rule</h3>
  <p>Separate content this way and you never write the same thing twice.</p>

  <div className={styles.layers}>
    <div className={styles.layer}>
      <div className={styles.num}>1</div>
      <h4>Shared knowledge</h4>
      <p>Lives in <code>references/</code>, <code>docs/</code>, <code>templates/</code>. Any agent reads it. Don't duplicate.</p>
    </div>
    <div className={styles.layer}>
      <div className={styles.num}>2</div>
      <h4>Workflows (skills)</h4>
      <p>Same basic <code>SKILL.md</code> shape in two folders: <code>.claude/skills/</code> for Claude, <code>.agents/skills/</code> for Codex. Sync the files, then adapt tool-specific details if needed.</p>
    </div>
    <div className={styles.layer}>
      <div className={styles.num}>3</div>
      <h4>Tool-specific config</h4>
      <p>Stays in <code>.claude/</code> and <code>.codex/</code>. These don't overlap. Don't try to merge them.</p>
    </div>
  </div>
</div>

## 7. The fastest way to convert a Claude project

Open the Claude-built project in Codex and paste this prompt. Codex can create the adapter files for you.

<pre className={styles.promptBox}>{`I built this project in Claude Code and want it to also work well in Codex.

Please inspect the project and create a Codex adapter setup.

Do these things:

1. Create AGENTS.md at the project root.
   - Use CLAUDE.md as the source of project knowledge.
   - Do not duplicate long sections unnecessarily.
   - Explain that AGENTS.md is the Codex-facing adapter.
   - Include a clear project map.

2. Create .codex/config.toml.
   - Start with a minimal safe config.
   - Do not add secrets.

3. Create .agents/skills/.
   - Copy important Claude skills from .claude/skills/ into .agents/skills/.
   - Keep each SKILL.md and supporting files together.
   - Do not put skills in .codex/skills/.

4. Create .codex/agents/ for any important Claude agents.
   - Claude agents are .md files in .claude/agents/.
   - Codex agents are .toml files in .codex/agents/.
   - Convert the agent instructions into developer_instructions.

5. Update .gitignore if needed.
   - Keep local overrides and secrets out of git.

Before editing, show me the files you plan to create or change.`}</pre>

<div className={styles.callout}>
  <p><strong>The whole conversion idea:</strong> do not rebuild the project. Add a Codex layer beside the Claude layer.</p>
</div>

## 8. How to maintain it

The setup only works if both tools stay in sync. These are the habits that matter.

<div className={styles.explainer}>
  <p><strong>When you update project instructions:</strong> keep <code>CLAUDE.md</code> and <code>AGENTS.md</code> aligned. Don't maintain two full copies. Pick one as the source of truth and make the other an adapter.</p>
  <p><strong>When you create a Claude skill:</strong> also copy it to <code>.agents/skills/</code> if Codex should use it.</p>
  <p><strong>When you create a Claude agent:</strong> also create a matching <code>.codex/agents/&lt;name&gt;.toml</code> if Codex should spawn it.</p>
  <p><strong>When you add shared knowledge:</strong> put it in <code>docs/</code>, <code>references/</code>, or <code>templates/</code> so every tool can read the same source.</p>
</div>
